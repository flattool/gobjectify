## Signals
GObject signals allow for a GObject to broadcast that something happened, which may be reacted to outside of the GObject.

## Background - Signals in Plain GJS
Declaring custom signals in plain GJS requires manual configuration within the `GObject.registerClass` call, and results in un-typed, raw-string-based signals at runtime.
```ts
class MyBox extends Box {
	static {
		GObject.registerClass({
			GTypeName: "MyBox",
			Signals: {
				"item-selected": {
					param_types: [Gtk.ListItem.$gtype, GObject.TYPE_DOUBLE],
				},
				"selection-cleared": {},
			},
		}, this)
	}

	select_item(index: number): void {
		// pretend we have a `get_list_item` method
		const item: Gtk.ListItem = this.get_list_item(index)
		this.emit("item-selected", item, index)
	}

	clear_selection(): void {
		// pretend we have a selection object
		this.selection.clear()
		this.emit("selection-cleared")
	}
}

const mb = new MyBox()

// No typing for arguments. `item` and `index` are both `any`
mb.connect("item-selected", (_box, item, index) => {
	print(`Item selected at index: ${index}`)
})

// No in-editor / compile-time error for a typo of a signal name
//             v  Missing 'e'
mb.connect("selction-cleared", () => {
	print("Selection was cleared!")
})
```
Every user-provided signal is untyped, and does not show up in GJS's signal list when using GIR typing. There's no compile-time error for incorrect argument types or misstyped signal names.
